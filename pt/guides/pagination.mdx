Todos os endpoints de listagem na Vendaze API retornam resultados paginados. Nenhum endpoint retorna uma lista ilimitada de registros.

## Como funciona

Use os parâmetros `page` e `per_page` para navegar pelos resultados:

```bash
GET /v1/tasks?page=1&per_page=50
GET /v1/tasks?page=2&per_page=50
```

## Metadados de paginação

Toda resposta de listagem inclui um objeto `meta`:

```json
{
  "data": [ ... ],
  "meta": {
    "total": 234,
    "page": 2,
    "per_page": 50,
    "has_more": true
  }
}
```

| Campo      | Descrição                                               |
| ---------- | ------------------------------------------------------- |
| `total`    | Total de registros que correspondem aos filtros atuais. |
| `page`     | Número da página atual.                                 |
| `per_page` | Número de registros por página.                         |
| `has_more` | `true` se existem mais páginas após esta.               |

Para calcular o total de páginas: `Math.ceil(meta.total / meta.per_page)`.

## Parâmetros disponíveis

| Parâmetro  | Tipo    | Padrão       | Máximo | Descrição                                                                        |
| ---------- | ------- | ------------ | ------ | -------------------------------------------------------------------------------- |
| `page`     | integer | `1`          | -      | Número da página a buscar.                                                       |
| `per_page` | integer | `50`         | `100`  | Registros por página. Retorna `422` se acima de 100.                             |
| `order_by` | string  | `created_at` | -      | Campo de ordenação. Os valores aceitos variam por recurso. Veja a tabela abaixo. |
| `order`    | string  | `desc`       | -      | `asc` ou `desc`.                                                                 |

## Buscar todos os registros

Itere pelas páginas até `has_more` ser `false`:

```javascript
async function fetchAll(endpoint, accessToken) {
  const results = [];
  let page = 1;

  while (true) {
    const res = await fetch(`https://api.vendaze.com${endpoint}?page=${page}&per_page=100`, {
      headers: { Authorization: `Bearer ${accessToken}` },
    });

    if (!res.ok) {
      const { error } = await res.json();
      throw new Error(`${error.code}: ${error.message}`);
    }

    const { data, meta } = await res.json();
    results.push(...data);

    if (!meta.has_more) break;
    page++;
  }

  return results;
}
```

## Calculando o total de páginas

```javascript
const totalPages = Math.ceil(meta.total / meta.per_page);
console.log(`Página ${meta.page} de ${totalPages}`);
```

## Considerações de performance

O valor de `per_page` afeta tanto a latência quanto o throughput. Escolha com base no seu caso de uso:

**Use `per_page` menor (10-25) quando:**

- Renderizando resultados em uma UI onde o usuário vê a primeira página imediatamente
- Você precisa de baixa latência na primeira resposta
- Você está paginando de forma interativa e a maioria dos usuários nunca passa da página 2

**Use `per_page` maior (50-100) quando:**

- Buscando todos os registros para uma sincronização ou exportação
- Executando um job em background que processa registros em lote
- Minimizar o número de chamadas à API importa mais do que o tempo de resposta individual

O máximo é `per_page=100`. Para exportações completas, use `per_page=100` e itere até `has_more` ser `false`.

<Note>
  Filtros são aplicados antes da paginação. `meta.total` sempre reflete a contagem de registros que
  correspondem aos seus filtros, não o total de registros no workspace.
</Note>

## Campos de ordenação por recurso

| Recurso           | Campos de ordenação                    |
| ----------------- | -------------------------------------- |
| Tarefas           | `created_at`, `updated_at`, `due_date` |
| Atividades        | `created_at`                           |
| Produtos          | `created_at`, `name`                   |
| Tags              | `created_at`                           |
| Listas            | `created_at`                           |
| Campos adicionais | `created_at`                           |

Passar um valor de `order_by` não suportado retorna `422` com uma mensagem listando os campos aceitos para aquele recurso.
