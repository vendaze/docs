A Vendaze API foi projetada com segurança em mente. Este guia descreve as práticas recomendadas para integradores que querem manter uma integração segura e confiável.

## Esta integração exige um back-end

A Vendaze API usa OAuth 2.1 com `client_secret`. Isso significa que sua integração **precisa ter um componente server-side**: um back-end que você controla, onde os segredos são armazenados e as trocas de token acontecem.

Esta não é uma integração para:

- Single-page applications (SPAs) que rodam inteiramente no browser
- Aplicativos mobile sem um back-end dedicado
- Sites estáticos ou front-ends serverless

Se seu aplicativo não tem back-end hoje, você precisa construir um antes de integrar com a Vendaze. O `client_secret` nunca pode ser enviado para nenhum ambiente que o usuário final possa inspecionar.

<Warning>
  Qualquer `client_secret` encontrado no DevTools do browser, em um binário de app mobile ou em um
  repositório público deve ser tratado como comprometido. Rotacione imediatamente.
</Warning>

## Como você recebe o client_secret

O `client_secret` nunca é retornado em nenhuma resposta da API. Ele é entregue exclusivamente por e-mail, via um link seguro que expira em 15 minutos.

**No registro:** após chamar `POST /v1/auth/register-app`, você recebe um e-mail com um link. Abra-o em até 15 minutos para visualizar seu `client_secret`. Copie-o para seu gerenciador de segredos imediatamente. O link não pode ser reaberto após expirar ou após você visualizá-lo.

**Na rotação:** após chamar `POST /v1/auth/rotate-app`, um novo link é enviado para o e-mail cadastrado no aplicativo. A mesma janela de 15 minutos se aplica. O `client_secret` anterior é invalidado no momento em que você envia a requisição de rotação, não quando você abre o link.

Se o link expirar antes de você abri-lo, envie uma nova requisição de rotação. Um novo link será enviado.

<Warning>
  Se você perder o `client_secret` e o link tiver expirado, você precisa rotacionar suas
  credenciais. Use `POST /v1/auth/rotate-app` com seu `client_id` e e-mail cadastrado para receber
  um novo link.
</Warning>

## Proteja o client_secret

Trate o `client_secret` como uma senha de banco de dados:

- Armazene em um gerenciador de segredos (AWS Secrets Manager, HashiCorp Vault, Cloudflare Secrets). Nunca em variáveis de ambiente que aparecem em logs ou artefatos de build.
- Nunca inclua em repositórios de código, mesmo que privados
- Nunca exponha em frontend, app mobile ou qualquer cliente que rode no dispositivo do usuário
- Rotacione imediatamente se suspeitar de comprometimento

## Armazene tokens com segurança

Access tokens e refresh tokens concedem acesso aos dados do workspace do seu usuário:

- Armazene refresh tokens criptografados em banco de dados, nunca em cookies ou localStorage
- Use `httpOnly` e `Secure` para cookies de sessão que referenciem tokens
- Implemente expiração e rotação de refresh tokens. A Vendaze API invalida o refresh token anterior a cada renovação.
- Revogue tokens imediatamente quando o usuário desconectar a integração via `POST /oauth/revoke`

## Use HTTPS em todos os redirects

O `redirect_uri` registrado no seu app deve sempre usar `https://`. A única exceção é `localhost` durante desenvolvimento. Em produção, qualquer redirect sem HTTPS é rejeitado com erro `invalid_redirect_uri`.

## Valide o parâmetro state

O parâmetro `state` no fluxo OAuth protege contra ataques de CSRF. Seu uso correto:

```javascript
// Antes de redirecionar o usuário para /oauth/authorize
const state = randomUUID();
sessionStorage.setItem('oauth_state', state);

// Na callback, antes de processar o code
const receivedState = new URL(window.location).searchParams.get('state');
const expectedState = sessionStorage.getItem('oauth_state');

if (receivedState !== expectedState) {
  throw new Error('State mismatch: possible CSRF attack');
}
```

## Solicite apenas os escopos necessários

Registre e solicite apenas os escopos que sua integração realmente precisa. Escopos desnecessários ampliam a superfície de ataque e reduzem a confiança dos usuários na hora de autorizar.

| Se você precisa de...         | Solicite...                                                                      |
| ----------------------------- | -------------------------------------------------------------------------------- |
| Apenas ler contatos           | `people:read`                                                                    |
| Criar e atualizar negócios    | `deals:write`                                                                    |
| Sincronização completa de CRM | `people:read people:write companies:read companies:write deals:read deals:write` |

## Rotacione o client_secret regularmente

Use `POST /v1/auth/rotate-app` para rotacionar suas credenciais a qualquer momento. Após a rotação:

1. Um novo link é enviado por e-mail para o endereço cadastrado no aplicativo. Abra-o em até 15 minutos para visualizar o novo `client_secret`.
2. O `client_secret` anterior é invalidado imediatamente quando você envia a requisição de rotação.
3. Tokens existentes permanecem válidos até expirarem naturalmente.
4. Novas trocas de code e renovações de refresh token exigem o novo `client_secret`.

## Monitore o uso da API

Use os headers `X-RateLimit-Remaining` e `X-Request-ID` para monitorar padrões de uso. Picos inesperados podem indicar que tokens foram comprometidos. Nesse caso, revogue todos os tokens via `POST /oauth/revoke` e notifique seus usuários.

## Resposta a incidentes

Se suspeitar de comprometimento do `client_secret` ou de tokens:

1. Revogue todos os tokens ativos via `POST /oauth/revoke`
2. Entre em contato com o suporte em `security@vendaze.com`
3. Notifique os usuários afetados
4. Rotacione suas credenciais via `POST /v1/auth/rotate-app`
5. Solicite nova autorização aos usuários

<Warning>
  Nunca implemente a troca do authorization code no frontend. O `client_secret` nunca deve ser
  exposto em código que rode no cliente.
</Warning>
